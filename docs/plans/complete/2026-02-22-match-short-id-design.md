# Match Public `short_id` Design

## Context
- Current public match URLs expose internal UUID (`matches.id`).
- Internal relations, mutations, and cache keys are UUID-centric.
- Requirement: external URL should use short public identifier while internal PK remains UUID.

## Goals
- Keep `matches.id` (UUID) as internal PK/FK.
- Add `matches.short_id` as public identifier.
- Canonicalize public routes to `/m/{short_id}`.
- Keep legacy `/matches/{uuid}` and `/matches/{uuid}/manage` compatible via permanent redirects.

## Non-goals
- Replacing all internal UUID usage in mutations/joins.
- Reworking team exercise routes in this change.

## Approach Options

### Option A (Recommended): Dual Identifier
- Add `short_id` column with DB-level uniqueness.
- Accept `id | short_id` in read paths.
- Generate links using `short_id` (fallback to UUID for safety).
- Keep mutations on UUID only.

Pros:
- Lowest regression risk.
- Minimal invasive changes to existing data model.
- Clean migration path with backward compatibility.

Cons:
- Temporary coexistence of two identifiers in code paths.

### Option B: Full Public-ID Rewrite
- Replace all route params and downstream code with `short_id` only.
- Convert mutation paths to resolve `short_id` first.

Pros:
- Single external identifier model.

Cons:
- High blast radius.
- Larger chance of mutation/query cache regressions.

### Option C: Keep UUID URLs + Add Alias
- Keep `/matches/{uuid}` as canonical and optionally add `/m/{short_id}`.

Pros:
- Minimum implementation effort.

Cons:
- Does not satisfy requirement to hide UUID in public URL.

## Selected Design
Use **Option A**.

### Data Model
- Add `public.matches.short_id TEXT`.
- `NOT NULL`, `UNIQUE`, base62 fixed length (10 chars).
- Default generated by `pgcrypto`-backed function.
- Backfill existing rows before enforcing `NOT NULL`.

### Read/Write Behavior
- Read API (`getMatchDetail`) resolves by UUID or `short_id`.
- Mutation APIs continue to require UUID.
- UI DTOs expose both internal id (`matchId`/`id`) and public id (`publicId`).

### Routing
- New canonical routes:
  - `/m/[identifier]`
  - `/m/[identifier]/manage`
- Legacy routes:
  - `/matches/[id]` -> 308 redirect to `/m/{resolved-short-id}`
  - `/matches/[id]/manage` -> 308 redirect to `/m/{resolved-short-id}/manage`

### Error Handling
- Redirect resolver falls back to raw identifier when DB lookup fails.
- Detail/manage pages continue showing existing not-found UI when match lookup fails.

### Verification
- Type check: `npx tsc --noEmit`
- Lint check: `npm run lint`
- Route/path grep for `/matches/{id}` usages in match guest/host flows.
