# DRAFT PRD (Product Requirements Document) v1.0

> **"더 많은 사람이 농구를 접하고, 한국 농구가 확대되는 세상"**
>
> 분산된 농구 커뮤니티를 하나로 모아, 누구나 쉽게 팀을 알리고 경기에 참여할 수 있는 플랫폼

---

## 1. 문제 정의 (Problem Definition)

### 왜(Why) 이 제품을 만드는가?

#### 우리의 비전
**"더 많은 사람이 농구를 접하고, 한국 농구가 확대되는 것"**

- 더 많은 사람이 자신의 팀을 알리고
- 더 많은 사람이 체육관을 빌려서 농구하고
- 결과적으로 한국 아마추어 농구 문화가 성장하는 것

#### 현재 시장 상황

**한국 아마추어 농구의 형태**
- **길거리 농구**: 공원, 학교 코트에서 자유롭게
- **대회/리그**: 각종 아마추어 대회 참가
- **동호회**: 정기적으로 체육관 대관하여 운동

**분산된 커뮤니티 현황**
- **BDR 카페** (수도권 중심): 가장 큰 농구 커뮤니티, 용병 모집 게시판 활성화
- **지역 오픈채팅방**: 각 지역별 카카오톡 오픈채팅으로 운영
- **별도 커뮤니티**: 밴드, 소모임 앱 등 파편화된 채널들

→ **문제**: 농구인들이 여러 커뮤니티에 분산되어 있어, 하나의 플랫폼에서 모든 것을 해결할 수 없음

#### MVP 목표: BDR 카페의 불편함 해결

**BDR 카페를 1차 타겟으로 삼는 이유**
- 가장 큰 사용자 풀 보유 (수도권 농구인 집중)
- 용병 모집 문화가 이미 정착되어 있음
- 불편함이 명확하여 대안 제시 시 전환 가능성 높음

**1. 호스트(운영진)의 관리 피로** 🎯 핵심 해결 대상
- 수작업 모집 글 작성 → 댓글/문자로 신청 접수 → 엑셀로 명단 관리
- "입금 확인했나요?", "자리 남았나요?" 반복되는 문의
- 노쇼(No-Show) 관리, 연락두절 게스트 처리에 소모되는 시간
- 매주 같은 정보를 반복 입력하는 번거로움

**2. 게스트(신청자)의 불편함**
- 여러 커뮤니티를 돌아다니며 경기 찾기
- 인사 글 작성, 댓글 달기, 등업 등 불필요한 절차
- 실시간 잔여 자리 확인 불가, 이중 신청 위험
- 내가 신청한 경기 현황 한눈에 보기 어려움

**3. 팀 운영의 비효율**
- 카톡 단톡방에서 매주 참석 여부 확인
- 읽씹하는 팀원들, 불명확한 인원 파악
- 인원 부족 시 급하게 다른 커뮤니티에서 용병 모집

#### 우리의 해결 전략

```
1단계 (MVP): 게스트 모집 → BDR 카페 사용자 확보
2단계: 팀 시스템 → 동호회 운영 효율화, 게스트 모집과 연계
3단계+: 지역 확장 → 지방 커뮤니티까지 통합
```

#### 비즈니스적 근거

- **시장 규모**: 국내 농구 동호인 약 50만명 (대한농구협회 추정)
- **경쟁 우위**: 농구 특화 플랫폼 부재 (축구는 플랩풋볼, 풋살매치 등 존재)
- **1차 타겟**: BDR 카페 활성 사용자 → 이들이 전환하면 자연스럽게 확산
- **수익 모델**: (Phase 3+) 프리미엄 기능, 체육관 제휴, 장비 광고

## 2. 대상 사용자 및 시나리오 (Target User & Scenarios)

### 1차 타겟: BDR 카페 사용자

BDR 카페에서 활동하는 **호스트**와 **게스트**가 MVP의 핵심 타겟이다.
이들의 불편함을 해결하면 자연스럽게 입소문을 통해 확산될 것으로 기대한다.

### 핵심 사용자 (Personas)

#### Persona A: 호스트 김민수 (32세, 직장인) 🎯 **MVP 최우선 타겟**
- **배경**: 5년차 동호회 "강남 슬램" 운영진
- **현재 행동**: 매주 BDR 카페에 모집 글 작성, 카톡으로 입금 확인, 노션으로 명단 관리
- **Pain Point**:
  - 퇴근 후 피곤한데 용병 관리에 1-2시간 소요
  - "입금했어요" 메시지 확인하느라 개인 카톡 알림 폭탄
  - 노쇼 당하면 다시 급하게 사람 구해야 함
- **Goal**: "글 등록부터 확정까지 1분, 나머지는 시스템이 처리"

#### Persona B: 게스트 이수진 (28세, 대학원생)
- **배경**: 학부 때 농구부, 졸업 후 운동 기회 감소
- **현재 행동**: 가끔 BDR 카페 눈팅, 등업이 귀찮아서 신청 포기한 적 있음
- **Pain Point**:
  - 어디서 경기를 찾아야 할지 모름
  - "저 신청해도 될까요?"라는 심리적 장벽
  - 시간대/지역 맞는 경기 찾기 어려움
- **Goal**: "비행기 예매하듯 원클릭으로 농구 신청"

#### Persona C: 팀장 박준영 (35세, 자영업자)
- **배경**: "역삼 파이터즈" 팀 창설 멤버, 주 1회 정기 운동
- **현재 행동**: 매주 카톡 단톡방에 참석 여부 확인, 부족하면 BDR에서 용병 모집
- **Pain Point**:
  - "이번 주 참석!" 카톡 읽씹하는 팀원들
  - 인원 부족할 때마다 급하게 BDR 글 작성
  - 회비 걷는 것도 일일이 확인
- **Goal**: "팀원 출석 체크 자동화, 부족하면 바로 게스트 모집"

### 핵심 사용자 여정 (User Journeys)

#### Journey 1: 호스트의 경기 등록

```
1. 앱 접속 → 2. [경기 등록] 버튼 → 3. 날짜/시간/장소 입력
→ 4. 포지션별 인원 설정 → 5. 참가비/계좌 입력 → 6. [등록 완료]
→ 7. 신청 알림 수신 → 8. [수락/거절] 원클릭 처리
```

#### Journey 2: 게스트의 경기 참가

```
1. 앱 접속 → 2. 지역/날짜 필터 → 3. 경기 리스트 탐색
→ 4. 상세 정보 확인 → 5. [신청하기] 버튼 → 6. 포지션 선택 + 신청
→ 7. 수락 알림 대기 → 8. 확정 후 참가
```

#### Journey 3: 팀 운동 생성 및 게스트 전환

```
1. 팀 페이지 접속 → 2. [이번 주 운동 생성] 버튼 (정보 자동 입력)
→ 3. 팀원에게 알림 발송 → 4. 팀원들 [참석/불참/미정] 투표
→ 5. 참석 인원 확인 → 6. 부족 시 [남은 자리 게스트 모집] 버튼
→ 7. 게스트 모집 시작 (메인 리스트에 노출)
```

## 3. 해결 방안 및 기능 명세 (Solution & Functional Requirements)

### 3.1 MVP 기능: 게스트 모집 시스템 (Phase 1) ✅ 구현 완료

#### 3.1.1 매치 리스트 (Guest View)
- **User Story**: "농구할 곳을 찾는 게스트로서, 내 조건에 맞는 경기를 빠르게 찾고 싶다. 왜냐하면 시간과 지역이 맞아야 신청할 수 있기 때문이다."
- **기능**:
  - 지역/날짜/포지션 필터링
  - 날짜별 그룹핑 (오늘, 내일, 이번 주)
  - 실시간 모집 상태 표시 (모집중/마감임박/마감)
  - 무한 스크롤 리스트

#### 3.1.2 매치 상세 (Match Detail)
- **User Story**: "경기 신청을 고민하는 게스트로서, 상세 정보를 한눈에 보고 싶다. 왜냐하면 시설, 비용, 분위기를 알아야 결정할 수 있기 때문이다."
- **기능**:
  - 핵심 정보 요약 (시간, 장소, 비용, 주차)
  - 호스트 공지사항 (Chat Bubble 스타일)
  - 카카오맵 연동 (지도보기, 길찾기)
  - 모집 현황 표시 (포지션별 확정 인원)
  - 공유 기능 (카카오톡, 링크 복사)

#### 3.1.3 매치 생성 (Host Create)
- **User Story**: "바쁜 호스트로서, 최소한의 입력으로 경기를 등록하고 싶다. 왜냐하면 매주 같은 정보를 반복 입력하기 싫기 때문이다."
- **기능**:
  - 간편한 폼 (날짜, 시간, 장소, 비용, 포지션)
  - 카카오맵 주소 검색
  - 시설 정보 입력 (주차, 샤워, 코트 크기)
  - 공지사항 작성 (선택)
  - 연락처 설정 (전화번호 또는 오픈카톡)

#### 3.1.4 신청/수락 시스템
- **User Story**: "호스트로서, 신청자를 빠르게 확인하고 처리하고 싶다. 왜냐하면 답변이 늦으면 다른 경기로 가버리기 때문이다."
- **기능**:
  - 게스트: 포지션 선택 후 원클릭 신청
  - 호스트: 신청자 목록 실시간 업데이트
  - 호스트: 수락/거절 원클릭 처리
  - 양측: 상태 변경 알림 (앱 내 알림)

#### 3.1.5 내 일정 (Schedule)
- **User Story**: "활동적인 게스트로서, 내가 신청한/확정된 경기를 한눈에 보고 싶다. 왜냐하면 일정을 관리해야 이중 신청을 피할 수 있기 때문이다."
- **기능**:
  - 참여 예정 매치 (게스트로 확정된 경기)
  - 관리 중 매치 (호스트로 등록한 경기)
  - 신청 대기/거절 히스토리
  - 진행 상태별 필터 (예정/완료/취소)
- **상태값 정의**:
  - **게스트 모집 (GUEST_MATCH)**:
    | 탭 | 상태 | 설명 |
    |----|------|------|
    | 게스트 | 신청 대기 | 호스트 승인 대기 중 |
    | 게스트 | 참여 예정 | 호스트가 승인 완료 |
    | 게스트 | 거절됨 | 호스트가 거절 |
    | 호스트 | 모집중 | 게스트 신청 받는 중 |
    | 호스트 | 모집 마감 | 모집 완료 |
  - **팀 운동 (TEAM_MATCH)** (Phase 2):
    | 탭 | 상태 | 설명 |
    |----|------|------|
    | 게스트 | 참여 예정 | [참석] 투표 완료 |
    | 게스트 | 불참 | [불참] 투표 완료 |
    | 게스트 | 미정 | [미정] 투표 완료 |
    | 호스트 | 투표중 | 팀원 투표 진행 중 |
    | 호스트 | 투표 마감 | 투표 종료, 인원 확정 |
    - Note: 투표는 **팀 탭**에서 진행, 투표 완료 시 경기관리에 표시

### 3.2 확장 기능: 팀 시스템 (Phase 2) 🚧 개발 예정

#### 3.2.1 팀 생성 (Team Create)
- **User Story**: "동호회 운영진으로서, 우리 팀을 등록하고 팀원들을 관리하고 싶다. 왜냐하면 정기 운동 인원 관리가 힘들기 때문이다."
- **기능**:
  - 팀명, 팀 소개(짧은 설명) 입력
  - 팀 표시 코드 입력
  - 홈구장(또는 활동 지역) 입력
  - 정기 운동 시간 설정 (요일/시간) - 매치 생성 시 자동 입력용
  - 회비 계좌 정보 입력 (팀 내부 회비 관리용, 게스트 참가비와 별개)
  - 팀 로고 업로드 (선택)
- **Trigger**: 생성 즉시 작성자는 `Team_Members` 테이블에 `ROLE='OWNER'`, `STATUS='ACCEPTED'`로 등록

#### 3.2.2 팀 초대 (Team Invite)
- **User Story**: "팀장으로서, 기존 팀원들을 쉽게 초대하고 싶다. 왜냐하면 일일이 가입하라고 말하기 번거롭기 때문이다."
- **기능**:
  - "초대 링크 복사" 기능
  - 링크 형태: `draft.kr/teams/invite/{team_uuid}`
  - 유효기간 설정 (선택, 기본 무제한)
- **Note**: 별도 "팀 검색" 기능은 MVP에서 제외

#### 3.2.3 팀 가입 (Team Join)
- **User Story**: "초대받은 사용자로서, 링크를 타고 바로 팀에 가입하고 싶다. 왜냐하면 복잡한 절차 없이 바로 팀 활동에 참여하고 싶기 때문이다."
- **기능**:
  - 초대 링크 접속 시 팀 정보 확인 화면
  - [팀 가입하기] 버튼 클릭
  - `Team_Members`에 `STATUS='PENDING'`으로 생성
  - 팀장에게 가입 신청 알림

#### 3.2.4 팀원 관리 (Team Management)
- **User Story**: "팀장으로서, 팀원 가입을 승인하고 역할을 관리하고 싶다. 왜냐하면 누가 팀원인지 명확히 해야 운영이 가능하기 때문이다."
- **기능**:
  - Owner 전용 관리 페이지
  - 가입 대기자 승인(`ACCEPTED`) / 거절
  - 기존 팀원 강퇴
  - 매니저 등급 부여 (일부 권한 위임)
- **역할 체계**:
  - OWNER: 모든 권한
  - MANAGER: 매치 생성, 가입 승인
  - MEMBER: 투표 참여

#### 3.2.8 팀 프로필 (Team Profile)
- **User Story**: "팀에 관심 있는 사용자로서, 팀 정보를 확인하고 싶다. 왜냐하면 어떤 팀인지 알아야 가입 여부를 결정할 수 있기 때문이다."
- **기능** (Phase 2 기본):
  - 팀 이름, 로고, 짧은 소개
  - 활동 지역
  - 정기 운동 시간 (요일/시간)
- **역할별 팀 페이지 뷰**:
  | 역할 | 볼 수 있는 것 | 할 수 있는 것 |
  |-----|-------------|-------------|
  | **게스트** (비팀원) | 팀 프로필 (이름, 소개, 활동 지역) | [가입 신청] 버튼 |
  | **팀원** | 팀 정보 + 공지사항 + 다가오는 운동 | 투표 참여, 회비 납부 현황 확인 |
  | **팀장/관리자** | 전체 | 운동 생성, 가입 승인, 팀원 관리, 공지 작성, 회비 체크 |
- **향후 확장** (Phase 2+):
  - 팀 매치 기록/결과 표시
  - 팀 활동 통계 (총 매치 수, 팀원 수 등)

#### 3.2.5 팀 매치 생성 (Team Match)
- **User Story**: "팀장으로서, 정기 운동을 원클릭으로 생성하고 싶다. 왜냐하면 매주 같은 정보를 입력하기 싫기 때문이다."
- **기능**:
  - [이번 주 운동 생성] 버튼
  - 팀 생성 시 입력한 정보 자동 불러오기:
    - 날짜: 이번 주 정기 운동 요일로 자동 설정
    - 시간: 정기 운동 시간으로 자동 설정
    - 장소: 홈구장 정보 자동 입력
    - 계좌: 게스트 모집 전환 시 사용할 수 있도록 별도 설정
  - `Matches` 생성 (`match_type = 'TEAM_MATCH'`)
  - 팀원 전체에게 알림 발송
- **경기관리 연동**:
  - **팀장/매니저** (호스트 탭): "투표중" 상태로 표시 - 팀원들의 투표 현황 확인 가능
  - **팀원** (게스트 탭): 투표 완료 시에만 표시 (참여 예정 / 불참)
  - 투표는 **팀 탭**에서 진행 → 투표 시 `Applications` 레코드 생성 → 경기관리에 반영
  - 투표 마감 후: 참석자는 "참여 예정", 불참자는 "불참"으로 표시
- **Note**: 자동 매치 생성 기능(스케줄 기반)은 MVP에서 제외, 수동 생성만 지원

#### 3.2.6 출석 투표 시스템 (Voting)
- **User Story**: "팀원으로서, 이번 주 참석 여부를 쉽게 표시하고 싶다. 왜냐하면 카톡에서 묻어가기가 싫고 명확히 표시하고 싶기 때문이다."
- **투표 위치**: 팀 탭 (팀 페이지 내 다가오는 운동 섹션)
- **기능**:
  - 팀원에게 [참석] / [불참] / [미정] 버튼 노출
  - 투표 클릭 시 `Applications` 레코드 생성 (`source = 'TEAM_VOTE'`)
    - [참석]: `STATUS='CONFIRMED'`
    - [불참]: `STATUS='NOT_ATTENDING'`
    - [미정]: `STATUS='PENDING'`
  - 불참/미정 선택 시 사유 입력 가능 (`description` 필드)
  - 실시간 투표 현황 표시 (참석 N명 / 불참 N명 / 미정 N명 / 미응답 N명)
- **투표 마감**: 팀장/매니저가 [투표 마감] 버튼으로 수동 마감
  - 마감 전까지 자유롭게 투표 변경 가능 (불참 → 참석 등)
  - 마감 후 투표 변경 불가
  - 미응답자는 미응답 상태로 유지 (별도 application 레코드 없음)

#### 3.2.7 게스트 모집 전환 (Guest Recruitment Switch)
- **User Story**: "팀장으로서, 인원이 부족할 때 바로 게스트를 모집하고 싶다. 왜냐하면 경기를 취소하기 싫고 외부 인원으로 채우고 싶기 때문이다."
- **기능**:
  - 투표 현황판: `(총원 12명) - (참석 7명) = 5자리 남음`
  - [남은 자리 게스트 모집하기] 버튼 (팀장/매니저에게만 노출)
  - 클릭 시 `recruitment_setup` 업데이트
  - 메인 리스트(게스트 모집)에 노출 시작
- **제어 옵션**:
  - 모집 포지션 선택 (팀 부족 포지션 자동 선택)
  - 게스트 모집 인원 제한
  - 참가비 설정: 팀 회비 계좌 사용 또는 별도 계좌 입력 선택 가능
- **게스트에게 노출되는 팀 정보**:
  - 기본 노출: 팀 이름, 팀 로고, 짧은 소개
  - 클릭 시: 팀 프로필 전체 공개 (정기 운동 시간, 활동 지역 등)
- **Note**: 팀 매치에서 게스트 모집 전환 시, 메인 게스트 모집 리스트에 노출됨
  - 추후 팀 매치 전용 리스트/탭 분리 예정 (Phase 2+)

#### 3.2.9 출석 통계 (Attendance Stats)
- **User Story**: "팀장으로서, 팀원들의 참석률을 한눈에 보고 싶다. 왜냐하면 누가 활동적인지 파악해야 팀 운영에 참고할 수 있기 때문이다."
- **기능**:
  - 팀원별 참석률 표시 (예: "최근 10경기 중 8회 참석 - 80%")
  - 팀 관리 페이지에서 전체 팀원 참석률 목록 조회
  - 팀원 프로필에 개인 참석률 표시
- **데이터 소스**: `applications` 테이블의 기존 투표/참석 데이터 활용
- **계산 기준**: 최근 N경기 (기본값 10) 기준

#### 3.2.10 회비 관리 (Fee Management)
- **User Story**: "팀장으로서, 팀원들의 월 회비 납부 현황을 관리하고 싶다. 왜냐하면 누가 냈는지 일일이 기억하기 어렵기 때문이다."
- **기능**:
  - 월별 회비 납부 체크리스트
  - 팀원별 [납부 완료] / [미납] 상태 토글
  - 납부 현황 요약 (예: "12명 중 10명 납부")
  - 미납자 목록 조회
- **Note**: 실제 결제 연동 없이, 팀원이 입금하고 팀장은 그걸 확인하는 형식
- **데이터 모델**: `team_fees` 테이블 추가 필요
  ```sql
  team_fees (
    id, team_id, user_id,
    year_month VARCHAR(7),  -- '2026-02'
    is_paid BOOLEAN DEFAULT false,
    paid_at TIMESTAMP,
    updated_by UUID  -- 체크한 사람 (팀장/매니저)
  )
  ```

#### 3.2.11 팀 공지사항 (Team Announcements)
- **User Story**: "팀장으로서, 팀원들에게 중요한 공지를 전달하고 싶다. 왜냐하면 카톡에서 묻히는 경우가 많기 때문이다."
- **기능**:
  - 팀 페이지에 공지사항 작성 (팀장/매니저)
  - 공지 목록 표시 (최신순, 고정 공지 지원)
  - 팀원에게 새 공지 알림 (앱 내 알림)
- **사용 예시**:
  - "이번 주 장소 변경: A체육관 → B체육관"
  - "다음 주 운동 휴무 안내"
  - "신규 팀원 OOO님 환영합니다"
- **데이터 모델**: `team_announcements` 테이블 추가 필요
  ```sql
  team_announcements (
    id, team_id, author_id,
    title VARCHAR(100),
    content TEXT,
    is_pinned BOOLEAN DEFAULT false,
    created_at, updated_at
  )
  ```

## 5. 범위 및 제약 사항 (Scope & Constraints)

### MVP 범위 (Phase 1) - 완료

**포함 (In Scope):**
- 게스트 모집 매치 CRUD
- 신청/수락/거절 플로우
- 지역/날짜/포지션 필터링
- 매치 상세 정보 (시설, 지도, 공유)
- 내 일정 관리
- 기본 사용자 프로필

**제외 (Out of Scope):**
- 팀 시스템 전체
- 인앱 결제
- 평점/리뷰
- 푸시 알림 (앱 내 알림만)
- 채팅 기능

**포함:**
- 팀 생성/초대/가입

- 팀원 관리 (역할, 승인, 강퇴)
- 팀 매치 생성 (정보 자동 입력)

- 게스트 모집 전환
- 다중 팀 가입 지원
- **출석 통계** (팀원별 참석률)
- **회비 관리** (월별 납부 체크리스트)
- **팀 공지사항** (공지 작성/조회)

**제외:**
- 팀 검색 기능 (초대 링크만)

- 자동 결제 연동 (회비는 수동 체크만)

## Appendix: 데이터 모델 설계 (Phase 2)

### Teams 테이블

```sql
CREATE TABLE teams (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  name VARCHAR(50) NOT NULL,
  short_intro VARCHAR(100),  -- 게스트에게 노출되는 짧은 소개 (예: "강남에서 매주 화요일 저녁 농구!")
  description TEXT,          -- 팀 프로필 상세 설명
  logo_url VARCHAR(500),
  home_court_name VARCHAR(100),
  home_court_address VARCHAR(200),
  region_code VARCHAR(20),
  regular_day VARCHAR(10),   -- 'MON', 'TUE', ... (매치 생성 시 자동 입력용)
  regular_time TIME,         -- 매치 생성 시 자동 입력용
  -- 팀 내부 회비 관리용 계좌 (게스트 참가비와 별개)
  fee_account_bank VARCHAR(20),
  fee_account_number VARCHAR(30),
  fee_account_holder VARCHAR(20),
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);
```

### Team_Members 테이블

```sql
CREATE TABLE team_members (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  team_id UUID REFERENCES teams(id) ON DELETE CASCADE,
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  role VARCHAR(20) DEFAULT 'MEMBER',  -- 'OWNER', 'MANAGER', 'MEMBER'
  status VARCHAR(20) DEFAULT 'PENDING',  -- 'PENDING', 'ACCEPTED', 'REJECTED'
  joined_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(team_id, user_id)  -- 같은 팀에 중복 가입 방지, 다른 팀에는 중복 가입 가능
);
```

**Note**: 한 사용자가 여러 팀에 소속될 수 있음 (예: 회사 동호회 + 동네 동호회)

```sql
  is_paid BOOLEAN DEFAULT false,
  paid_at TIMESTAMP,
  updated_by UUID REFERENCES users(id),  -- 체크한 사람 (팀장/매니저)
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(team_id, user_id, year_month)
);
```
